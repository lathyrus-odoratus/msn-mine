### 2. 技術選型討論

**18:02 開發者**（喊停直接動工）：我想我們至少先討論技術選型，可以嗎？

**討論**：拆成三個層面——

  伺服器持有狀態最自然；Supabase Realtime 是 broadcast 性質、沒有權威端，
  防作弊和回合仲裁會很彆扭。
- 前端：Vanilla（最輕）vs Vite+React vs Vite+Vue。
- 部署：本機/區網 vs 公網（Vercel 不能跑 WebSocket）。

**決定**：Node + ws 自架（權威伺服器）、Vite + Vue（開發者偏好 Vue）、
部署目標是開發者自己的 GCP VM。（POC 於 18:02–18:25 間完成並通過測試）

## 2026-06-28 第一層開工：邀請、觀戰、嗆聲、可選場地

> 本節由 **Claude Opus 4.8（1M context）** 接手協作。這次 session 未擷取每句話的
> 精確時間戳，故本節按發生順序記錄、不標時間（不臆造時間以維持「真實時間戳」原則）。

### 10. 下一步選什麼

**開發者**：問專案下一步有哪些選項。

**Claude**：對照 roadmap，目前卡在第一層。按投入/回報排序：馬上能做且回報高的是
遊戲說明、結果分享；中等的是音效動畫、路由、自訂旗、chat；偏基礎建設的是對局紀錄。

**開發者**：我想做這批——遊戲說明、音效、動畫、路由邀請、自訂旗子、嗆聲、對局紀錄、觀戰。
你的看法？

**判斷**：方向認同（只有觀戰跨到第二層）。但這批依賴差很多，關鍵洞察是
**「對局事件流」是隱形共用核心**——現在每步算完廣播就丟掉、不留歷史。提了兩個提醒：
對局紀錄別急著上 Postgres（與第一層「零營運負擔」相衝，輕量版先寫 JSONL/SQLite）；
觀戰是這批最重、又跨層，建議當壓軸。

### 11. 遊戲狀態記在哪 → 重播能不能推後

**開發者**：目前遊戲狀態記在哪裡？

**Claude**：全在伺服器記憶體三層（game 物件 / rooms Map / 連線上），**沒有任何歷史**，
重連靠的是 `snapshot()` 重建當前盤面、不是重播。所以對局紀錄/觀戰/replay 都缺同一塊：
按順序記每一手的事件流。

**開發者**：那可以先做觀戰、進來就讀當前局面，把重播先推後？

**關鍵判斷**：可以，而且更省——**即時觀戰根本不需要事件流 log**。`snapshot()` 已存在
（本來給重連用），觀戰者「進來讀當前局面」要的就是它。即時觀戰用「當前狀態」、重播才需
「每一手順序」，兩者要的資料不同，所以重播能乾淨推後、不卡觀戰。

### 12. 觀戰的設計決策

**開發者**：選 B（先做路由 `/room/ABCD` 再觀戰）。觀戰者清單要可見嗎？可以嗆聲嗎？

**判斷**：
- 人數顯示給全房（臨場感、被觀看的壓力即樂趣）；名字看是否讓觀戰者報名。
- **嗆聲沒有作弊疑慮**——關鍵安全檢查：觀戰者看到的資訊和玩家完全一樣（只有已翻開的格），
  看不到未翻開的雷，所以喊話洩漏不了情報。治理問題（洗版、對陌生人開放）是第三層的事。

**開發者**：觀戰者一開始給菜市場名字、可隨時改；嗆聲做兩三種就行。問實作順序。

**Claude**：四項是一條**嚴格依賴鏈**（嗆聲要名字、名字要觀戰者實體、觀戰者要入口路由），
順序被鎖死：路由 → 觀戰唯讀+人數 → 菜市場名字+改名 → 嗆聲，且每步都能獨立 ship。

### 13. 「旁觀沒用，點了會變進房間」

**開發者**：回報觀戰連結點了會變成進房當玩家。

**判斷**：先用 e2e 與正式站三人重現，**證明伺服器是對的**（第 2 人當玩家、第 3 人才觀戰）。
所以這不是 bug 是**設計漏洞**——房間沒坐滿時，連結會把人補進空座位，沒有「主動觀戰」入口。
**修法**：加 `spectate` 訊息與大廳「加入對戰／觀戰」兩個明確選項，座位空著也能只看不玩。

### 14. 場地大小可選

**開發者**：幫我做個地圖小一點的版本：12×12、29 雷。

**Claude**：先問是「直接取代」還是「建房時可選」。指出前端已是尺寸無關（棋盤/計分都吃
伺服器送來的 width/height/winTarget），所以可選版本不難。

**決定**：建房時可選（標準 16×16 / 小場 12×12・29 雷・先搶 15），房間記住自己的尺寸、
觀戰與重連都跟著走。

## 2026-06-30 音效、房間列表、AI 強化

### 21. 音效

**開發者**：來做音效。

**判斷**：用 Web Audio API 即時合成、不打包音檔——POC 輕量又合復古 MSN 味，事件直接掛在
既有 ws 訊息流（搶雷分自己/對手、換手、勝負、嗆聲、開局）。靜音開關存 localStorage。
旁觀者終局給中性音，不按勝負（呼應後面第 23 節同一個「旁觀不該有勝負視角」的判斷）。

### 22. 房間列表與路徑

**開發者**：加房間清單，首頁放「進行中(n)」連結點進列表、可上一步回首頁；replays(/:id)、
playing 也給個路徑。

**判斷**：「進行中」定義為已開打且未分勝負的房間（才有觀戰價值）。導入輕量前端路由
（/rooms、/replays/:id），回放改吃網址 → 可分享、可重新整理、瀏覽器上一步自然可用；
對局中畫面仍由遊戲流程主導，route 只管大廳子畫面。

### 23. 旁觀畫面與電腦補強

**開發者**：旁觀不該顯示「你輸了」；想把電腦邏輯補強。

**判斷**：旁觀者沒有「你」的視角 → 結束只報勝方、不給「再來一局」。
電腦補強：原 greedy 只搶 100% 確定雷、其餘亂猜；改用機率推理（smart）——對每格估是雷的
機率、每手挑最高，能結合多線索避開安全格。因為「搶到雷得分又續手」，挑高機率會滾雪球，
模擬對 greedy 先手勝率約 98%。沿用既有「bot 一改版本就標進紀錄」原則，smart 列為新 bot。

## 2026-06-30 新玩法腦力激盪：探雷模式（Mine Runner）

> 完整規格見 `docs/explore-mode.md`，此處只記決策與判斷。

### 25. 新模式的兩個承重決定

**開發者**：想做 3D／固定斜角、角色在地圖上奔跑找雷，three.js 合適嗎？

**判斷**：「用什麼畫」是風險最低的決定，真正的分岔是**互動模型**。釐清後開發者選
**回合制移動 + 固定斜角 2.5D** → 這組剛好最貼合現有引擎：回合制可沿用權威伺服器與
對局事件流，2.5D 不必養 three.js 資產管線（原型先 CSS 斜角，質感再上 PixiJS）。
當成獨立新模式，不動經典對戰。

### 26. 先收斂再發想

**開發者**：列出玩家操作——移動、標記、檢查、踩 0 傳送、調查、道具。要繼續發想還是收斂？

**判斷**：清單自己裂成「核心動詞（移動/標記/揭露）」與「道具/能力系統」兩半，而道具正是
roadmap 標紅的「每加一個組合空間翻倍」無底洞 → **先收斂核心、道具打包進 Phase 2**。
關鍵前提：本作「踩到雷是好事」，風險是反的（浪費行動值在空格），這決定了各動詞的意義。

### 27. 讓移動有價值：標記→揭露 + 自由分配行動值

**開發者**：要翻地必須先標記再揭露；玩家自由分配行動值。

**判斷**：標記從「記憶輔助」升級成「揭露的必經前置」，有了機制重量；行動值池天然吸收
經典的「踩雷續手」。**承重柱**：標記/揭露必須限角色射程內，否則移動無價值。

### 28. 標記是公開意圖、但被迷霧鎖住（a+b 自平衡）

**開發者**：標記要親臨；揭露不限視野一次打開、被偷就失效；對手只在 n 格內看得到你的標記，
離開後標記不變但不知動態（不知被偷）；已翻格公開。偷雷對稱，揭露失敗也是合理情報。

**判斷**：(a 先標後翻) 與 (b 標記近距公開) 撞出**自我平衡的心理戰**——早標記＝佈局但曝險、
當場翻＝安全但放棄規劃。收斂出乾淨原則「**事實公開、意圖迷霧**」。代價是伺服器要按玩家
各自算可見視圖（唯一真正新工程），但 Replays／觀戰維持上帝視角。

### 29. 行動值＝時間（甲案）＋ 時間軸排序 ＋ 防雪球

**開發者**：選「甲」（行動值與時間同一貨幣）；回合順序用 Thebes／拼布對決的時間機制
（時間用少的一方一直行動）；XP→等級→拉高行動上限；喜歡後追，要更多防一面倒的機制。

**判斷**：時間機制本身就是免費的「節奏型後追」。但 **XP→上限→更多 XP 是富者愈富的雪球**
（時間軸平衡頻率、平衡不了吞吐量）→ **XP/升級延後，原型用固定上限**。防雪球通則：
**補貼落後者會誘發裝弱**，故原型只放「假裝不來」的兩個——領先者更曝光（重用迷霧）＋
時間軸節奏追趕；補貼型/變數型留作備案。
