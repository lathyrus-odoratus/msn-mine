## 2026-06-11 開案日

### 1. 起點：想做 MSN 對戰踩地雷

**18:00 開發者**：我想要做個 msn 對戰踩地雷網頁小遊戲，你對玩法熟悉嗎？

**Claude**：熟悉，是 Minesweeper Flags——跟單人踩地雷相反，不是避雷而是**搶雷**：
16×16 棋盤埋 51 顆雷（奇數不會平手），輪流點擊，點到雷得 1 分且續手，
點到空格換對手，先搶 26 顆獲勝。數字是雙方共享的情報。

**決定**：對戰模式選「線上即時對戰優先，AI 對手等 POC 跑通再做」。

### 2. 技術選型討論

**18:02 開發者**（喊停直接動工）：我想我們至少先討論技術選型，可以嗎？

**討論**：拆成三個層面——
- 即時通訊層：自架 Node+ws vs Supabase Realtime vs Cloudflare Durable Objects。
  關鍵判斷：**回合制對戰需要權威仲裁者**（誰的回合、是不是雷、分數），
  伺服器持有狀態最自然；Supabase Realtime 是 broadcast 性質、沒有權威端，
  防作弊和回合仲裁會很彆扭。
- 前端：Vanilla（最輕）vs Vite+React vs Vite+Vue。
- 部署：本機/區網 vs 公網（Vercel 不能跑 WebSocket）。

**決定**：Node + ws 自架（權威伺服器）、Vite + Vue（開發者偏好 Vue）、
部署目標是開發者自己的 GCP VM。（POC 於 18:02–18:25 間完成並通過測試）

### 3. POC 完成後：部署前要做什麼

**18:25 開發者**：git init。跟我說要部署還要做哪些事。

**Claude**：列了部署清單（上 VM、防火牆、systemd、nginx、固定 IP），
並指出程式層面唯一值得部署前先補的是**斷線重連**——上公網後手機鎖屏、
網路抖動造成的斷線會是最常見的體驗殺手，當時的實作是一斷線房間就解散。

**18:26 開發者**：請補上。
**決定**：補斷線重連（token + 60 秒寬限期 + 棋盤快照同步）。

### 4. 基礎設施揭露與部署

**18:39 開發者**：跟你說說我的 GCP VM 配置——用 Docker、domain name 用
Cloudflare、server 在 ssh wisp。

**勘查後的判斷**：
- VM 的模式是「每服務一容器綁 localhost port + Cloudflare Tunnel 按 hostname
  分流到 `*.miao-bao.cc`」，VM 不開 80/443，新服務照同樣模式接最自然。
- **Cloudflare 會切斷閒置 100 秒的連線**——回合制遊戲很容易發呆超過 100 秒，
  所以必須加 WebSocket ping keepalive（每 30 秒），這是因部署環境而生的必要修正。

**問**：子網域取什麼？ **答**（開發者）：要有對戰的意象。
**決定**：`mineduel.miao-bao.cc`（duel = 一對一決鬥）。18:53 建立 DNS，
隨後上線並通過正式環境 WebSocket 對戰驗證。

### 6. 遊戲八角理論（Octalysis）

**19:09 開發者**：你知道遊戲八角理論嗎？

**Claude**：周郁凱的 Octalysis，八個核心驅動力 + 白帽/黑帽維度。
拿它映射 roadmap：覆蓋意外地均勻（成就→排行榜、創意授權→道具天賦、

### 7. 整體 review：會不會太重？

**19:14 開發者**：你幫我整個 review 一下，會不會太重？

**判斷**：
- 清單裡藏著兩個不同的產品：前半是「好玩的小遊戲」，後半（登入、DB、Elo、
  代幣任務）是「營運中的 live service」——後者不是功能而是**承諾**。
- 最大風險不是做不完，是**在還沒有人可留的時候先蓋留存系統**。
- 道具/天賦/場地的隱形成本在平衡與測試，每加一個道具組合空間翻倍。
- 建議不砍清單、加閘門：三層結構，沒過門檻不動工。最壞情況是做完第一層
  就膩了，損失幾個週末，留下的還是完整好玩的遊戲。

### 8. 登入與紀錄之辯

**19:16 開發者**：不過沒做登入不就丟失了遊戲紀錄了嗎？

**Claude**：會丟的只有「這是誰打的」，不會丟「打了什麼」——**紀錄和身分可拆**：
- 對局紀錄掛在 game ID 上，replays/觀戰/分享零登入就能做到位。
- 個人戰績用 localStorage 匿名 ID（Wordle 的做法），朋友圈階段夠用。
- 登入真正買到的只有跨裝置同步、排行榜公信力、防小號——第三層才值錢。
- 匿名先行的順路好處：日後登入時認領匿名 ID，歷史戰績無縫接上。

**19:17 開發者**：恩。
**決定**：「對局紀錄儲存・輕量版」提前到第一層（從現在開始攢數據），
登入留在第三層。原則定為**紀錄先行、登入殿後**，寫入 README roadmap。
